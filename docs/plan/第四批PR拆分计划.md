## 一、拆分原则

第四批 PR 聚焦**交互体验创新与系统完善**。

第四批 PR 的目标不是功能补全（第二批已完整），也不是成本控制（第三批已落地），而是在完全可用的系统基础上，通过创意设计提升用户体验，彰显项目的设计思考和工程美感。

每个 PR 仍遵循单一功能原则，且**完全不影响主链路** ——出问题直接回滚，零风险。

## 二、第四批 PR 列表

### PR16：Avatar 状态联动——AI 角色可视化展示

**标题**：`[体验创新] 实现 Avatar 状态联动，可视化展示 AI 角色`

**功能描述**：
在界面中加入一个可爱的 AI Avatar（小人或宠物），与系统四大状态（待机、录音、推理、播报）深度绑定，实时展示当前系统状态，并在语音播报期间随语速节奏做出相应的身体/表情变化，提升对话的沉浸感与趣味性。

**设计思路**：
- Avatar 采用**帧序列动画**实现，每个状态 3～5 帧 PNG 图像。
- 素材风格：像素风或卡通化简笔，便于多帧制作。
- 状态到动作的映射：
  - `IDLE`：待机晃动（循环 3 帧，每帧 200ms）。
  - `RECORDING`：侧耳倾听（循环 4 帧，耳朵颤动，眼睛闭合）。
  - `THINKING`：思考状态（循环 3 帧，头顶打问号，身体微动）。
  - `SPEAKING`：语音播报（循环 4 帧，嘴巴张合，身体随律动节奏摇摆）。
- 在 `ui/` 目录下新增 `avatar_widget.py`，定义 `AvatarWidget` 类，订阅全局状态机，实时切换动画帧。
- 语音播报期间，`AvatarWidget` 订阅 TTS 的音频数据，根据音量幅度调整摇摆幅度（可选但推荐实现）。

**实现方案细节**：

1. **素材管理**
   - 在 `assets/avatar/` 目录下按状态分类存放 PNG 帧图：`idle/`、`recording/`、`thinking/`、`speaking/`。
   - 每个目录内 frame_0.png、frame_1.png ……
   - 建议分辨率 256×256，透明背景（PNG alpha 通道）。

2. **AvatarWidget 实现**
   - 继承 `QLabel`，使用 `QTimer` 驱动帧切换。
   - 订阅 `app/state.py` 中的全局状态信号，状态变化时切换动画序列。
   - 支持鼠标悬停时显示状态提示文字（"正在录音中……"）。

3. **与 UI 布局的融合**
   - Avatar 放在对话区上方或右侧，不与对话历史冲突。
   - 建议大小 150×150～200×200。

4. **与 TTS 的可选联动**
   - 若要实现"随音量摇摆"效果，需要 `services/tts_client.py` 暴露当前播放的音量数据接口。
   - 这个部分是可选的，不是 PR16 的必须项，可作为 PR16 完成后的后续优化。

**测试方式**：
- 启动程序后 Avatar 显示待机动画。
- 按下录音后 Avatar 切换到侧耳倾听动画，连贯无卡顿。
- ASR 识别中 Avatar 切换到思考动画。
- TTS 播报期间 Avatar 切换到播报动画，口型变化明显。
- 状态切换时无黑屏、无闪烁、无内存泄漏。
- 关闭 Avatar（假设后续有开关）不影响程序运行。
- **合并后，主分支代码可正常启动运行，核心功能无回归问题。**

**影响范围**：
- `ui/avatar_widget.py`（新增）
- `ui/main_window.py`（集成 Avatar 组件）
- `app/state.py`（确保状态信号暴露）
- `assets/avatar/`（新增素材目录）

**前置依赖**：
- PR15 UI 布局完成后推进（需要有清晰的空间位置放 Avatar）。
- 素材必须事先准备好，建议提前 1-2 天。

**合规同步动作**：
- README 补充 Avatar 功能说明与设计思路。
- 在"设计亮点"或"交互体验"章节里写明 Avatar 的创意设计与实现意义。
- 在代码注释和设计文档中说明"Avatar 状态联动"是对标准状态栏的创新可视化"。

## 四、素材选择建议

建议三种方向：

1. **像素风宠物**（推荐）
   - 网站：itch.io、OpenGameArt 有大量免费像素风素材
   - 特点：文件小、易于手工调整、演示时观感统一
   - 建议：下载现成的"Idle / Walk / Attack"动画包，提取适用帧

2. **卡通简笔**（技术难度低）
   - 用 AI 图生（如 DALL·E / Midjourney）生成，再手工简化为帧序列
   - 特点：独特，容易彰显项目特色
   - 建议：生成一个"可爱的 AI 助手"的设定图，再手工切分为状态

3. **代码绘制矢量**（风险较高）
   - 用 `QPainter` 手绘，完全程序化
   - 特点：零素材成本，纯技术实现
   - 建议：除非美术功底强，否则效果可能不理想

**推荐方案**：选择现成的像素风宠物素材（如"像素猫"、"像素狗"等网络素材），提取 4 个状态各 4 帧，合计 16 张 PNG，总文件大小不超过 1MB。

## 五、与赛事评审的关系

### 5.1 是否算"原创功能"

**算**。Avatar 状态联动不在赛题基础要求中，属于主动设计的交互创新。

在设计文档和答辩时可以强调：
- "为了提升用户沉浸感，我们在标准状态栏之外，设计了一个与系统状态深度绑定的 AI Avatar，实现了状态的可视化与拟人化表达。"
- "这体现了对用户体验和交互设计的考量，不仅仅是功能完成，更是设计思考。"

### 5.2 评分影响

- **功能分**：无影响（功能已在 PR1～PR15 完成）。
- **创新分**：+小分（取决于评审对"交互创新"的看重程度，通常有帮助但不是核心）。
- **演示分**：+中分（Avatar 动画使演示更生动、更吸引评审注意力）。
- **风险**：极低（完全独立，不影响主链路）。

### 5.3 答辩时的表述

建议在答辩时这样说：
> "在完成基础功能后，我们思考了如何进一步提升用户体验。传统的文字状态栏显示信息量受限，我们设计了一个 AI Avatar，通过动画和拟人化表情实时反映系统状态，使对话过程更具沉浸感，同时也彰显了设计思考。"
