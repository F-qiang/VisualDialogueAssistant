# 第二批 PR 拆分计划

## 一、拆分原则

第二批 PR 在第一批主链路闭环（视觉输入、语音输入、ASR、文本对话、上下文管理）的基础上，按照"多模态增强、体验提升、系统健壮"的原则推进。

每个 PR 聚焦一个独立增强目标，不破坏已有主链路，主分支随时可运行。

**前置工作（PR6 开始前完成）**：将 `app/state.py` 中的状态枚举定义提前落地，作为全局基础设施：

```python
class AppState(str, Enum):
    IDLE = "idle"
    RECORDING = "recording"
    THINKING = "thinking"
    SPEAKING = "speaking"
```

PR6～PR10 直接引用，避免各 PR 各自临时写状态字符串、后续统一时大量返工。

### PR6：VLM 接入与视觉问答闭环

**标题**：`[推理层] 接入 VLM 并打通视觉问答闭环`

**功能描述**：
实现 `VLMClient`，将摄像头当前帧与用户问题一起发送给视觉语言模型，获取带有图像理解能力的回答，完成"摄像头帧 + 语音问题 → 视觉回答"的闭环。

**实现思路**：
- 实现 `services/vlm_client.py` 中的 `VLMClient.chat_with_image`，调用 VLM API（如 GLM-4V / Qwen-VL）。
- 在调用前使用 `core/vision.py` 中的 `compress_frame` 压缩图像，统一压缩至**宽度 640px、JPEG 质量 70%**，确保单图体积可量化（可在文档中写明「单图体积减少约 70%」）。
- 在图像上传前新增两个轻量画质检测函数（均基于 OpenCV 原生实现，各约 5 行）：
  - `check_brightness(frame)`：计算平均灰度值，低于阈值（如 40）判定为「光线偏暗」，直接提示用户无需上传。
  - `check_sharpness(frame)`：使用拉普拉斯方差算法，低于阈值判定为「画面模糊」，直接提示用户无需上传。
- 画质检测通过后再调用 VLM，对应设计文档里的端侧前置过滤策略。
- 将历史对话记录拼接到请求中，支持多轮视觉追问。
- 在 `core/vision.py` 中预留 `motion_detect()` 函数占位，方便第三批 PR 直接扩展，无需重构。

**测试方式**：
- 对着摄像头提问"画面里有什么"，能返回包含图像描述的回答。
- 光线过暗或画面模糊时，系统直接提示用户，不上传图像、不调用 VLM。
- 连续两轮带图追问时，回答能结合前一轮上下文。
- API 不可用时给出明确错误提示，不崩溃。
- **合并后，主分支代码可正常启动运行，核心功能无回归问题。**

**影响范围**：
- `services/vlm_client.py`
- `core/vision.py`（新增 `check_brightness`、`check_sharpness`，预留 `motion_detect` 占位）
- `core/pipeline.py`
- `.env.example`（新增 VLM 相关配置项）

**合规同步动作**：
- README 补充视觉问答功能说明。
- README 补充画质检测与成本控制说明（附压缩参数数据）。

### PR7：意图路由——自动判断文本问答与视觉问答

**标题**：`[对话层] 实现意图路由，自动切换文本与视觉问答`

**功能描述**：
完善 `core/router.py` 中的意图判断逻辑，使系统能够根据用户问题内容自动决定走纯文本 LLM 还是带图像的 VLM，节省 API 成本。

**实现思路**：
- 扩充 `judge_need_vision` 的视觉关键词列表，支持通过配置扩展，不需要改代码。
- 增加**上下文关联判断**：若上一轮对话走的是视觉路由，当前轮追问默认继续走视觉路由，避免"问桌上有什么→问它是什么颜色"因无关键词而错误走文本模型。
- 在 `core/pipeline.py` 中根据路由结果选择调用 `LLMClient` 或 `VLMClient`。
- 在路由模块预留缓存逻辑入口（函数签名占位），方便第三批 PR 实现请求缓存时直接扩展。
- 将路由决策结果记录到日志，方便调试和优化。

**测试方式**：
- 问"今天天气怎么样"走文本路由，不携带图像。
- 问"桌上有什么"走视觉路由，携带摄像头帧。
- 上一轮是视觉问答，追问"它是什么颜色"时，仍走视觉路由。
- 路由关键词通过配置项扩展，不需要修改代码。
- 两条路径切换时上下文不丢失。
- **合并后，主分支代码可正常启动运行，核心功能无回归问题。**

**影响范围**：
- `core/router.py`
- `core/pipeline.py`
- `app/config.py`（路由关键词配置项）

**前置依赖**：PR6 完成后再推进。

**合规同步动作**：
- README 补充意图路由机制说明（文本问答 vs. 视觉问答的切换逻辑）。

### PR8：TTS 接入与语音播报

**标题**：`[播报层] 接入 TTS 并实现语音播报`

**功能描述**：
实现 `TTSClient`，将 AI 回复文本合成为语音并自动播放，让系统从"文字回复"升级为"有声回复"。

**实现思路**：
- 实现 `services/tts_client.py` 中的 `TTSClient.synthesize`，调用 TTS API 合成音频并写入本地临时文件。
- 使用 `sounddevice` 或系统播放器播放生成的音频文件，播放完成后清理临时文件。
- 播放期间在界面状态栏显示 `AppState.SPEAKING` 状态提示。
- **播报互斥**：播放期间全局锁定麦克风采集，禁止录音，避免 TTS 声音被录入形成死循环。
- **打断机制**：支持用户按下指定快捷键（如空格键）或再次说话时立即终止播报，进入下一轮录音等待。
- 合成失败时降级为纯文字显示，不影响主流程。

**测试方式**：
- AI 回复后能自动合成并播放语音。
- 播放期间界面状态栏显示"正在播报"提示，不卡死。
- 播放期间麦克风不采集输入，TTS 声音不会被录入。
- 按快捷键后播报立即停止，进入下一轮录音等待。
- TTS API 不可用时降级为纯文字显示，主流程正常。
- **合并后，主分支代码可正常启动运行，核心功能无回归问题。**

**影响范围**：
- `services/tts_client.py`
- `utils/audio_utils.py`（新增播放与互斥锁工具函数）
- `ui/main_window.py`
- `.env.example`（新增 TTS 相关配置项）

**合规同步动作**：
- README 补充语音播报功能说明。
- README 补充 TTS 服务依赖与配置方法。

### PR9：全局状态管理与应用层整合

**标题**：`[应用层] 完善全局状态管理，整合各模块至应用主循环`

**功能描述**：
以 `app/state.py` 的状态枚举为基础，完善全局应用状态管理，将摄像头、音频、推理管道、TTS 等模块统一接入应用主循环，使系统从"各模块独立运行"升级为"完整联动的对话助手"。同时完善 `app/prompts.py` 中的系统提示词。

**实现思路**：
- 在 `app/state.py` 中扩展状态管理类，持有 `CameraService`、`DialoguePipeline`、`TTSClient` 等实例，统一管理全局标志。
- **并发请求锁**：在全局状态中增加"推理中"互斥锁，上一次请求未返回时新的语音输入不触发推理，避免并发请求导致状态错乱和回答乱序。
- 在 `app/prompts.py` 中补充针对视觉问答和纯文本问答的分别提示词模板。
- 通过 Qt 信号槽确保状态切换线程安全。
- `main.py` 在启动时初始化状态管理，形成完整的事件驱动主循环。

**测试方式**：
- 界面状态栏能实时反映当前阶段（录音 / 识别 / 推理 / 播报 / 空闲）。
- 推理期间连续说话，新的语音输入被正确丢弃，不出现乱序回答。
- 关闭程序后摄像头、音频等资源全部正常释放。
- 视觉问答和文本问答的回复风格符合各自提示词设定。
- **合并后，主分支代码可正常启动运行，核心功能无回归问题。**

**影响范围**：
- `app/state.py`
- `app/prompts.py`

- `ui/main_window.py`
- `core/pipeline.py`

### PR10：异常兜底与纯文本降级模式

**标题**：`[健壮性] 完善异常兜底与纯文本降级模式`

**功能描述**：
完善 `utils/error_handler.py`，为各关键路径（ASR、LLM、VLM、TTS、摄像头）统一添加异常捕获和用户友好的错误提示，并新增纯文本降级模式开关，保障演示稳定性。

**实现思路**：

- **纯文本降级模式**：新增一个布尔开关（可在界面操作或自动触发），网络异常时自动关闭摄像头、VLM 相关功能，保留基础文字问答，对应设计文档中的网络风险预案。

**测试方式**：

- 摄像头不可用时给出权限或设备提示。
- 网络异常自动切入纯文本降级模式后，文字问答功能仍正常工作。
- 异常恢复后可退出降级模式，恢复完整功能。
- 所有异常均有日志记录。
- **合并后，主分支代码可正常启动运行，核心功能无回归问题。**

**影响范围**：

- `services/asr_client.py`

### PR11：画面变化检测与视觉触发优化

**标题**：`[成本控制] 实现画面变化检测，减少重复视觉请求`

**功能描述**：
利用 `core/vision.py` 中预留的 `motion_detect()` 接口实现帧差法画面变化检测：仅在用户提问且画面发生明显变化时才截取新帧上传，相同画面不重复调用 VLM，降低 API 成本。

**核心实现**：
- 在 `motion_detect()` 中复用已有的 `detect_frame_change` 逻辑，对外暴露统一接口。
- 在 `core/pipeline.py` 的视觉路由路径中判断画面是否变化，变化则截新帧，否则复用上一帧描述。
- 在日志中记录"命中缓存"的次数，便于演示时展示成本控制效果。

**影响范围**：
- `core/vision.py`（实现预留的 `motion_detect`）
- `core/pipeline.py`

**合规同步动作**：
- README 补充画面变化检测与成本控制效果说明（附节省比例数据）。

### PR12：上下文摘要压缩与请求缓存

**标题**：`[成本控制] 实现上下文摘要压缩与请求缓存`

**功能描述**：
超出轮次上限的对话自动调用 LLM 进行摘要压缩，控制 Token 长度；短时间内相同问题 + 相同画面复用缓存结果，减少重复调用；补充调用次数统计日志，直观展示成本控制效果。

**核心实现**：
- 利用 `core/router.py` 中预留的缓存逻辑入口，实现简单的 LRU 缓存（问题文本 + 帧哈希为 Key）。
- 超出 `max_rounds` 时调用 LLM 对旧轮次生成摘要，替换原始记录写入上下文。
- 新增 `utils/logger.py` 的调用统计记录（总调用次数、VLM 调用次数、缓存命中次数）。

**影响范围**：
- `core/context.py`（摘要压缩）
- `core/router.py`（缓存实现）
- `utils/logger.py`（调用统计）

**合规同步动作**：
- README 补充摘要压缩与请求缓存说明，附调用统计示意数据。
