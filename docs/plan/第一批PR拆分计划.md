# 第一批 PR 拆分计划

## 一、拆分原则

本项目第一批 PR 按照“主链路优先、单功能推进、逐步可运行”的原则拆分。每个 PR 只覆盖一个明确目标，尽量保持变更范围小、验证方式清晰、主分支随时可运行。

第一批 PR 的目标不是一次性完成全部功能，而是先打通视觉输入、语音输入、AI 接入和上下文管理这几条核心链路，为后续视觉问答、语音播报和成本控制打下基础。

---

## 二、第一批 PR 列表

### PR1：项目骨架与基础配置

**标题**：`[初始化] 搭建项目骨架与基础配置`

**功能描述**：
完成项目目录结构、入口文件、基础配置文件和环境变量模板的初始化，为后续模块开发提供统一的工程基础。

**实现思路**：
- 创建 `app/`、`core/`、`services/`、`ui/`、`utils/` 等基础目录。
- 添加 `main.py` 作为程序入口。
- 补充 `.env.example`、`.gitignore`、`requirements.txt` 等工程文件。
- 初始化 README 的项目简介、技术依赖和运行说明。

**测试方式**：
- 检查目录结构是否创建完整。
- 确认程序能够从 `main.py` 启动。
- 确认 `.env.example`、`.gitignore`、`requirements.txt` 等文件内容正确。
- 检查 README 是否可以作为项目入口文档正常阅读。

**影响范围**：
- 项目根目录
- 配置层
- 文档层

**合规同步动作**：
- README 补充项目简介、技术依赖、项目边界和代码声明。
- 若新增第三方依赖，需在 README 与 `requirements.txt` 中同步说明。

---

### PR2：Qt 主窗口与摄像头预览

**标题**：`[采集层] 完成 Qt 主窗口与摄像头预览`

**功能描述**：
完成桌面端主窗口搭建，并接入摄像头实时预览功能，使系统具备最基础的视觉输入展示能力。

**实现思路**：
- 使用 `PySide6` 创建主窗口。
- 通过 `QTimer` 驱动界面刷新。
- 使用 `opencv-python` 打开摄像头并读取帧。
- 将摄像头画面渲染到 Qt 界面中。
- 预留状态栏或提示区域，用于后续显示录音、识别和推理状态。

**测试方式**：
- 启动程序后，主窗口可以正常打开。
- 摄像头能够成功启动并在界面中实时刷新。
- 关闭程序后摄像头资源能够正常释放。
- 摄像头权限不足时能够给出明确提示。

**影响范围**：
- `ui/`
- `core/camera.py`
- `main.py`

**合规同步动作**：
- README 补充摄像头预览功能说明。
- README 的技术依赖中补充 `PySide6` 与 `opencv-python`。

---

### PR3：语音输入与录音文件生成

**标题**：`[采集层] 完成语音输入与录音文件生成`

**功能描述**：
完成麦克风音频采集、静音检测和录音结束判定，使系统具备稳定的语音输入能力，并能输出可供 ASR 使用的音频文件。

**实现思路**：
- 使用 `sounddevice` 采集麦克风输入。
- 使用 `webrtcvad` 或轻量静音判断逻辑检测语音起止。
- 在用户停止说话后自动结束录音。
- 将音频保存为本地文件，作为后续语音识别输入。

**测试方式**：
- 对着麦克风说话时能够进入录音状态。
- 停止说话后录音能够自动结束。
- 录音文件能够正常生成并播放。
- 无有效语音输入时不会产生空录音文件。

**影响范围**：
- `core/audio.py`
- `utils/audio_utils.py`
- `ui/`

**合规同步动作**：
- README 补充语音输入功能说明。
- README 的技术依赖中补充 `sounddevice` 与 `webrtcvad`。

---

### PR4：ASR 接入与文本回复闭环

**标题**：`[推理层] 接入 ASR 并打通文本回复闭环`

**功能描述**：
接入语音转文字服务，将录音文件转换为文本，并基于文本生成第一版回复，完成“语音输入 → 文本识别 → 文本回答”的最小闭环。

**实现思路**：
- 封装 ASR API 请求。
- 将录音文件上传到语音识别服务。
- 解析返回结果并提取识别文本。
- 在识别成功后调用文本模型生成回复。
- 将识别文本和回复结果展示到界面中。

**测试方式**：
- 输入一段语音后能够成功识别为文本。
- 识别结果能够正确显示在界面中。
- 文本模型能够返回可读回复。
- 识别失败或置信度过低时有明确提示。

**影响范围**：
- `services/asr_client.py`
- `services/llm_client.py`
- `core/response.py`
- `core/pipeline.py`

**合规同步动作**：
- README 补充 ASR 服务依赖说明。
- README 补充文本对话模型依赖说明。

---

### PR5：上下文管理与多轮追问

**标题**：`[对话层] 实现上下文管理与多轮追问`

**功能描述**：
为系统增加对话上下文保存能力，支持多轮追问、历史信息引用和上下文裁剪，让系统从“单轮问答”提升为“连续对话助手”。

**实现思路**：
- 保存最近若干轮对话记录。
- 在请求模型时拼接最近上下文。
- 超出长度时裁剪旧消息或进行摘要压缩。
- 提供上下文清空接口，方便测试与演示。

**测试方式**：
- 连续输入两轮以上问题时，后续回答能结合前文内容。
- 清空上下文后系统不再引用旧信息。
- 上下文过长时能够自动裁剪，不影响主流程。

**影响范围**：
- `core/context.py`
- `core/pipeline.py`
- `services/llm_client.py`

**合规同步动作**：
- README 补充多轮对话说明。
- README 补充项目边界与当前完成阶段说明。

## 三、第一批 PR 的执行顺序建议

建议按照以下顺序推进：

1. PR1：项目骨架与基础配置
2. PR2：Qt 主窗口与摄像头预览
3. PR3：语音输入与录音文件生成
4. PR4：ASR 接入与文本回复闭环
5. PR5：上下文管理与多轮追问

该顺序的好处是：
- 先保证工程基础可运行。
- 再逐步打通输入链路。
- 然后完成核心推理闭环。
- 最后补齐连续对话能力。

## 四、第一批 PR 的命名与提交规范

### 4.1 PR 命名规范

建议使用以下格式：

`[模块] 功能简述`

示例：
- `[初始化] 搭建项目骨架与基础配置`
- `[采集层] 完成 Qt 主窗口与摄像头预览`
- `[推理层] 接入 ASR 并打通文本回复闭环`

### 4.2 Commit 命名规范

建议使用以下格式：

`[阶段/模块] 功能简述`

示例：
- `[第一阶段/初始化] 搭建项目骨架与基础配置`
- `[第一阶段/采集层] 完成摄像头预览`
- `[第二阶段/推理层] 接入语音识别服务`

### 4.3 文档同步规范

每个功能 PR 完成后，都应同步更新 README 对应章节，至少包括：
- 项目简介
- 技术依赖
- 运行说明
- 项目边界
- 代码声明

这样可以保证文档与代码始终一致，也符合赛事对持续交付和可追溯性的要求。

---

## 五、补充说明

第一批 PR 的目标不是追求功能大而全，而是优先完成主链路闭环，确保项目能够尽快进入“可运行、可演示、可持续迭代”的状态。后续在此基础上，再逐步补充多模态视觉理解、语音播报、成本控制与异常兜底等增强能力。
