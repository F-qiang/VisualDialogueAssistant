# AI 视觉对话助手项目设计文档

## 一、项目背景与目标

本项目面向“AI 视觉对话助手”赛题，拟实现一款可同时接入摄像头与麦克风的多模态对话应用。系统在运行过程中能够感知当前画面内容、接收用户语音输入，并结合视觉信息与上下文对话，生成自然、准确、可追问的回复。

本设计的目标不仅是完成基础功能，还需兼顾以下几个方面：

1. 视觉理解是否稳定、准确。
2. 语音交互是否自然、流畅。
3. 端侧与云端协同是否合理，是否能够控制运行成本。
4. 代码结构是否便于持续迭代、拆分 PR 与现场演示。
5. 是否满足赛事对原创性、持续提交和文档完整性的要求。

---

## 二、需求分析

结合赛事题目与提交规范，本项目需要满足以下核心要求。

### 2.1 功能要求

- 打开摄像头，获取并展示画面内容。
- 打开麦克风，接收用户语音。
- 将语音转换为文字并理解用户问题。
- 结合当前摄像头画面进行视觉分析。
- 返回恰当的文本回复，并支持语音播报。
- 支持多轮对话，能够进行上下文追问。

### 2.2 设计要求

- 需要考虑视觉内容理解准确性。
- 需要考虑语音交互的自然度与流畅性。
- 需要考虑端云协同的成本控制策略。
- 需要提交设计文档，说明计划实现的用户故事、最终实现情况，以及成本控制技巧。

### 2.3 合规要求

- 作品需自主完成，不得抄袭。
- 需要持续提交，不可临时突击。
- 每个 PR 尽量只做一件事。
- PR 描述应完整、清晰、可复现。
- 依赖和第三方库需在 README 中列明。
- 若复用历史代码，必须在 PR 中注明来源。

## 三、总体架构设计

本项目采用“采集展示层 → 前置过滤层 → 核心推理层 → 结果输出层”的四层结构。该结构边界清晰、职责单一，便于分模块开发、分 PR 提交和后续维护。

### 3.1 采集展示层

负责摄像头、麦克风和界面展示，只做输入采集与结果呈现，不承担复杂业务逻辑。

**职责：**
- 摄像头画面预览。
- 麦克风音频采集。
- 识别文本、视觉状态和回复结果展示。
- 语音播报播放。

**设计原则：**
- 不在这一层进行推理判断。
- 保持界面轻量、稳定、清晰。
- 仅保留基础输入输出职责。

### 3.2 前置过滤层

负责在请求进入大模型之前进行轻量判断，尽量减少无效调用，降低成本。

**职责：**
- 静音检测。
- 语音结束判定。
- 画面变化检测。
- 语音置信度判断。
- 输入有效性检查。

**设计原则：**
- 尽可能在端侧完成轻量判断。
- 无效输入直接拦截，不进入云端推理。
- 通过状态机控制流程，避免重复调用。

### 3.3 核心推理层

负责对用户输入进行理解、路由和回答生成，是系统的核心能力层。

**子模块：**

#### 3.3.1 意图路由子模块

根据用户问题判断是否需要视觉信息：
- 纯文本闲聊或知识问答，走文本模型。
- 涉及“画面、物体、桌面、手里、光线、距离”等词时，走多模态模型。

#### 3.3.2 多模态推理子模块

将用户语音转文字结果、当前关键帧图像和历史上下文一并送入模型，由模型生成回答。

#### 3.3.3 上下文管理子模块

维护最近若干轮对话，支持多轮追问。

**策略：**
- 只保留最近 5 轮完整对话。
- 超出部分做摘要压缩。
- 保持最新上下文完整，避免回答失焦。

### 3.4 结果输出层

统一处理回复展示和语音播报。

**职责：**
- 将模型输出展示到界面。
- 播放 TTS 语音回复。
- 输出异常提示和引导信息。

**设计原则：**
- 文本先展示，再进行播报。
- 播报期间暂停收音，避免回环。
- 对异常情况做分级提示。

## 四、功能模块设计

### 4.1 摄像头采集模块

**功能：** 实时获取摄像头画面，并在需要时截取关键帧作为视觉输入。

**设计要点：**
- 默认仅做预览。
- 只有在用户发起提问或检测到明显画面变化时才截帧。
- 传输前对图片进行压缩，降低带宽和调用成本。

**异常处理：**
- 摄像头开启失败时，提示用户检查设备权限。
- 画面模糊或光线不足时，提示用户调整位置或环境。

### 4.2 语音识别模块

**功能：** 将用户语音转换为文字。

**设计要点：**
- 使用静音检测判断一次语音输入是否结束。
- 设置语音置信度阈值，识别不可靠时要求用户重说。
- 空语音和无效语音直接丢弃，不触发后续流程。

### 4.3 意图识别与路由模块

**功能：** 判断用户问题是否需要视觉信息，并选择对应模型。

**设计要点：**
- 优先使用规则关键词做轻量判断。
- 不把所有问题都送入多模态模型。
- 降低推理成本，提高响应速度。

### 4.4 多轮上下文模块

**功能：** 维护最近对话内容，使系统具备连续对话能力。

**设计要点：**
- 保存最近 5 轮完整对话。
- 超出部分做摘要后再拼接。
- 支持一键清空上下文，便于演示切换。

### 4.5 回答生成模块

**功能：** 根据用户输入、视觉内容和上下文生成自然回复。

**设计要点：**
- 回复风格简洁、自然、口语化。
- 视觉问答先描述画面，再给出判断或建议。
- 控制单次输出长度，便于语音播报。

### 4.6 语音播报模块

**功能：** 将模型回复转成语音播放。

**设计要点：**
- 文本先展示，后播报。
- 长回复按句切分，避免卡顿。
- 播报失败时不影响文字结果展示。

## 五、数据流设计

系统数据流按照“采集 → 过滤 → 推理 → 输出”的顺序执行，整体闭环清晰，便于调试与演示。

### 5.1 标准流程

1. 系统启动，摄像头开始预览，麦克风进入待命状态。
2. 检测到有效语音后开始录音。
3. 语音结束后进行语音识别，得到文字结果。
4. 对识别结果做置信度判断：
   - 若失败，则提示用户重说，流程结束。
   - 若成功，则进入意图路由。
5. 意图路由判断是否需要视觉信息：
   - 纯文本问题：调用文本模型。
   - 视觉相关问题：截取当前关键帧并调用多模态模型。
6. 模型生成回答后，保存到上下文队列。
7. 将回答展示到界面，并进行语音播报。
8. 播报结束后，系统回到待命状态。

### 5.2 画面变化分支

如果系统检测到画面发生较大变化，但用户没有提问，则只输出轻量场景提示，不进入大模型推理，避免浪费调用次数。

## 六、成本控制策略

本项目的成本控制核心思想是：**把能够在端侧完成的工作尽量前置，只有必要时才调用云端模型**。

### 6.1 按需调用模型

- 纯文本问题不调用多模态模型。
- 只有视觉相关问题才带上图像调用模型。
- 无效输入直接拦截。

### 6.2 端侧过滤

- 使用静音检测减少无效录音。
- 使用画面变化检测避免重复上传帧。
- 使用置信度阈值减少低质量请求。

### 6.3 图片压缩

- 发送前压缩图片尺寸和质量。
- 降低网络带宽消耗。
- 减少云端处理成本。

### 6.4 上下文控制

- 只保留最近几轮对话。
- 旧上下文做摘要压缩。
- 控制 Prompt 长度，降低 token 消耗。

### 6.5 失败兜底

- 图像模糊时先提示用户调整画面。
- 语音识别失败时提示重说。
- 网络异常时仅保留本地提示，避免重复请求。

## 七、用户故事与实现目标

### 7.1 计划实现的用户故事

1. 作为用户，我希望能够直接通过语音向 AI 提问。
2. 作为用户，我希望 AI 能结合摄像头画面理解我在问什么。
3. 作为用户，我希望 AI 能记住上下文并支持多轮追问。
4. 作为用户，我希望在画面不清晰或光线不足时得到提示。
5. 作为开发者，我希望系统尽量减少不必要的模型调用，从而控制成本。

### 7.2 最终实现目标

- 实现语音输入与文字回复的基础链路。
- 实现摄像头画面理解能力。
- 实现视觉问答与文本问答的统一入口。
- 实现上下文对话能力。
- 实现语音播报与异常兜底。

## 八、开发迭代计划

### 8.1 第一阶段：基础底座

目标是保证项目先跑起来，并尽快形成可运行的第一版主链路。

- 摄像头预览。
- Qt 主窗口与基础布局。
- 麦克风采集。
- 语音识别。
- 纯文本问答。

### 8.2 第二阶段：视觉能力

目标是完成赛题核心能力，并补齐视觉相关的前置过滤逻辑。

- 截帧与图片压缩。
- 画面变化检测。
- 多模态推理接入。
- 视觉问答逻辑。

### 8.3 第三阶段：交互优化

目标是提升演示效果和稳定性，增强对话体验。

- 语音播报。
- 上下文摘要。
- 状态提示。
- 异常兜底。

### 8.4 第四阶段：成本优化与打磨

目标是让系统更稳定、更省钱、更好演示。

- 请求缓存。
- 并发控制。
- 模型路由优化。
- 界面优化。

### 8.5 第一批 PR 拆分计划

第一批 PR 以“主链路优先、单功能推进、逐步可运行”为原则，建议按以下顺序实施：

1. **PR1：项目骨架与基础配置**
   - 创建目录结构与入口文件。
   - 补充配置文件、环境变量模板和基础依赖说明。
   - 完成 README 的初始版本。

2. **PR2：摄像头预览与主窗口搭建**
   - 完成 Qt 主窗口。
   - 在界面中接入摄像头预览。
   - 先保证视觉输入链路可见、可测、可运行。

3. **PR3：语音输入与录音文件生成**
   - 完成麦克风采集。
   - 实现静音检测与录音结束判定。
   - 输出可用于 ASR 的音频文件。

4. **PR4：ASR 接入与文本回复闭环**
   - 接入语音转文字服务。
   - 打通文本回复主链路。
   - 完成“语音输入 → 文字识别 → 文本回答”的最小闭环。

5. **PR5：上下文管理与多轮追问**
   - 保存最近对话记录。
   - 实现多轮上下文拼接与裁剪。
   - 支持连续追问。

### 第一批 PR 的拆分依据

该拆分方式并非随意划分，而是基于以下三点：

1. **题目主链路驱动**：赛题核心链路是“看到画面、听到语音、生成回答、支持追问”，因此需要先打通输入、推理和上下文这四个关键环节。
2. **工程依赖顺序**：摄像头预览和语音输入是后续 AI 接入的前置条件；AI 接入是系统能否回答的核心；上下文管理建立在单轮回答可用的基础上。
3. **赛事提交规则**：赛事要求每个 PR 尽量只做一件事，且主分支需保持可运行状态，因此需要优先拆分成粒度清晰、可独立验证的小功能。

## 九、工程交付与版本管理设计

本项目不仅要完成技术功能，还需要满足赛事对持续提交、PR 规范、可追溯性和合规性的要求。因此，这一部分不是独立于技术设计之外的附加说明，而是对前面四层架构和分阶段迭代计划的交付保障设计。

### 9.1 PR 拆分原则

为匹配“采集展示层 → 前置过滤层 → 核心推理层 → 结果输出层”的分层架构，PR 拆分遵循“单模块、单功能”的原则，确保每个提交都可以独立验证、独立回滚、独立说明价值。

**对应关系如下：**
- 采集展示层相关能力，如摄像头预览、麦克风采集，可单独拆分为独立 PR。
- 前置过滤层相关能力，如静音检测、画面变化检测、语音置信度判断，可单独拆分为独立 PR。
- 核心推理层可按子模块继续拆分为意图路由、多模态推理、上下文管理三个方向的 PR。
- 结果输出层可拆分为文本展示、语音播报、异常提示等独立 PR。

这种拆分方式的目的，不只是满足赛事“每个 PR 只做一件事”的要求，也便于在开发过程中逐步验证主链路，降低一次性集成带来的风险。

### 9.2 PR 提交模板

为保证版本迭代过程可追溯、可复现、可评审，每个 PR 需要包含以下内容：

- **标题**：以“[模块] 功能简述”的形式描述变更，例如 `[采集层] 完成摄像头预览功能`。
- **功能描述**：说明该功能在整体架构中的位置、作用和使用方式。
- **实现思路**：简述对应模块采用的技术方案和核心逻辑。
- **测试方式**：说明如何验证该功能、如何检查异常场景。
- **影响范围**：说明本次修改影响到哪些模块、依赖或配置。
- **合规声明**：说明是否使用第三方库、是否存在历史代码复用。

该模板的设计目的是将 PR 作为设计文档的延伸说明，使每次提交都能够对应到具体模块和具体目标，便于答辩时追溯“为什么这样拆、为什么这样做”。

### 9.3 Commit 提交规范

提交信息采用统一格式：

`[阶段/模块] 功能简述`

**示例：**
- `[第一阶段/采集层] 完成摄像头预览功能`
- `[第二阶段/过滤层] 新增静音检测逻辑`
- `[第三阶段/推理层] 实现意图路由规则`

采用这种格式的原因是：
1. 便于从提交历史直接看出当前开发进度。
2. 便于将提交与分阶段迭代计划对应起来。
3. 便于后续整理 PR、撰写设计说明和答辩材料。

### 9.4 合规说明

赛事场景下，工程交付本身就是设计的一部分，因此需要在版本管理过程中同步满足合规要求：

- README 中应列出依赖、运行方式和项目边界。
- 使用第三方库时应明确说明库名、用途和作用模块。
- 若复用历史代码，需要在 PR 描述中注明来源。
- 项目应保持持续提交，避免最后一天集中提交导致作品有效性风险。

## 十、项目文档体系设计

为满足赛事的文档完整性、可追溯性与持续交付要求，README 不是独立的说明页，而是整个项目交付体系的重要组成部分。它需要和设计文档形成明确映射，共同说明项目目标、技术依赖、原创性、运行方式与边界范围。

### 10.1 README 的设计定位

README 建议承担以下角色：

1. **项目入口文档**：帮助评审或使用者快速理解项目是什么。
2. **依赖说明文档**：说明本项目依赖哪些第三方库和云端服务，以及各自用途。
3. **原创性说明文档**：说明项目中哪些部分属于本次开发的原创设计。
4. **运行指引文档**：说明项目如何安装、如何启动、如何验证。
5. **合规说明文档**：说明是否存在历史代码复用，以及如何满足赛事规范。

### 10.2 README 标准目录（可直接落地）

README 建议采用以下结构，便于直接编写和后续维护：

1. **项目简介**
   - 背景与目标
   - 核心功能概述
2. **快速开始**
   - 环境依赖说明（Python 版本、第三方库及用途）
   - 安装步骤
   - 启动命令
   - 功能验证方式
3. **技术设计**
   - 整体架构
   - 核心模块
   - 成本控制策略
4. **合规声明**
   - 原创性说明
   - 第三方依赖明细
   - PR / Commit 规范
5. **项目边界与风险兜底**
   - 已完成内容
   - 已知限制
   - 兜底方案

### 10.3 README 内容与设计模块的映射关系

README 的内容应尽量与前面的设计章节建立对应关系，避免成为单纯的信息罗列。

**建议映射如下：**
- **项目简介**：对应第一、二章中的项目背景与需求分析。
- **技术依赖**：对应第四、六章中的功能模块和成本控制策略。
- **原创功能说明**：对应第三、四、六章中的架构设计、模块设计和成本控制设计。
- **运行说明**：对应第五、十一章中的数据流设计和风险兜底方案。
- **项目边界说明**：对应第八章中的开发迭代计划，说明当前版本完成到什么程度。
- **代码声明**：对应合规要求，说明原创性与历史代码使用情况。

### 10.4 README 落地示例

为避免表述过于抽象，README 中建议直接写出具体依赖与用途示例，例如：

- `PySide6`：桌面界面、视频预览、状态展示。
- `opencv-python` + `numpy`：摄像头采集、画面预览、截帧与图像变化检测。
- `sounddevice` + `webrtcvad`：麦克风音频采集、静音检测与语音端点判断。
- 百度智能云 / 讯飞 ASR：语音转文字。
- 智谱清言 / 通义千问多模态 API：视觉问答推理。
- 智谱清言 / 通义千问文本模型：文本对话与上下文摘要。
- 百度智能云 / 讯飞 TTS：语音播报。

代码声明也建议给出明确模板，例如：

- **无复用场景**：本项目为本次赛事独立开发，未复用历史代码片段；第三方依赖明细与用途见“环境依赖”章节。
- **有复用场景**：本项目核心逻辑为本次赛事原创开发；其中某模块参考了公开实现思路，已在对应 PR 中注明来源；第三方依赖明细见“环境依赖”章节。

### 10.5 文档迭代节奏

README 需要与代码迭代同步更新：

- 完成每个阶段后，更新“快速开始”和“项目边界”。
- 接入新依赖或外部服务后，立即更新“环境依赖”和“合规声明”。
- 若存在代码复用或参考实现，必须在对应 PR 和 README 中同步说明。
- 文档更新应作为独立提交的一部分，并保持清晰可追溯。

### 10.6 设计文档与 README 的分工

- README 负责“怎么运行、有什么功能、有哪些依赖”。
- 设计文档负责“为什么这样设计、如何分层、如何控制成本、如何规避风险”。
- 两者内容保持一致，但侧重点不同，避免信息重复堆叠。

## 十一、风险与兜底方案

### 11.1 网络风险

如果云端接口不可用，系统至少应保留本地提示和文本模式，保证演示不中断。

### 11.2 设备风险

如果摄像头或麦克风权限不足，程序应提示用户检查权限，而不是直接退出。

### 11.3 识别风险

如果语音识别置信度过低，系统应要求用户重说，避免错误链路继续执行。

### 11.4 演示风险

现场演示优先保证主链路跑通：

- 语音提问。
- 语音转文字。
- 视觉问答。
- 文本与语音回复。

加分项可作为补充展示，不应影响主流程。

本项目采用四层架构和分阶段迭代方案，重点突出视觉理解、语音交互、上下文管理和成本控制四个方面。通过前置过滤、意图路由、上下文压缩和异常兜底，可以在保证功能完整性的同时降低运行成本，提高系统稳定性。
