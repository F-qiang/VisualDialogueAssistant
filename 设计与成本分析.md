# AI 视觉对话助手设计与分析

## 一、用户故事规划与最终实现

### 1. 计划实现的用户故事

1. 作为用户，我希望能通过摄像头和麦克风与系统自然对话。
2. 作为用户，我希望系统能根据我的提问判断是否需要看图。
3. 作为用户，我希望系统在画面模糊或光线过暗时给出提示，而不是盲目调用模型。
4. 作为用户，我希望系统能记住前文并支持多轮追问。
5. 作为用户，我希望系统能把回复朗读出来，形成完整语音闭环。
6. 作为用户，我希望界面能显示 Avatar，并根据状态变化给出反馈。
7. 作为用户，我希望能开启自动监听模式，减少手动点击。
8. 作为用户，我希望能看到状态反馈、成本统计与导出报告，便于调试和展示。

### 2. 最终实现的内容

1. 已实现摄像头预览、麦克风录音、ASR、LLM/VLM 调用与 TTS 回复。
2. 已实现基于规则的意图路由，能区分文本问题和视觉问题。
3. 已实现画质检测、视觉缓存、请求缓存、上下文摘要压缩等优化。
4. 已实现多轮对话、自动监听模式、状态反馈面板与成本统计面板。
5. 已实现 Avatar 浮层角色，支持状态同步、拖动、双击隐藏/显示。
6. TTS 语速和音量控制等辅助功能。

### 3. 未完全达成或尚待继续优化的内容

1. Avatar 的最终美术形象仍以精简版为主，后续可替换为更拟人化的角色素材。
2. TTS 播放链路在不同环境下仍可能有兼容性问题，后续需要继续稳定化。
3. 状态反馈面板目前更偏实用提示，后续可进一步做成更显眼的可视化卡片。
4. 长期记忆、复杂任务规划、多人对话等能力暂未重点覆盖。

## 二、控制运营成本的技巧与实际采用情况

### 1. 计划想到的成本控制技巧

1. 画面变化检测：只有画面变化明显时才考虑调用 VLM。
2. 画质前置过滤：在上传前检查亮度和清晰度，避免低质量图像进入模型。
3. 规则优先路由：优先用轻量规则判断是否需要视觉模型。
4. 视觉缓存：相同画面复用上一次的 VLM 结果。
5. 请求缓存：相同或高度相似的问题直接复用结果。
6. 上下文摘要压缩：将历史对话压缩为摘要，减少上下文长度。
7. 按需调用 TTS：只有需要语音播报时才合成音频。
8. 状态反馈与统计：通过统计面板观察调用次数和节省比例，及时调整策略。

### 2. 最终实际采用的成本控制措施

1. 已采用画面变化检测与画质检测，避免无效视觉请求。
2. 已采用视觉缓存和请求缓存，减少重复模型调用。
3. 已采用上下文摘要压缩，降低多轮对话的上下文成本。
4. 已采用规则优先路由，减少不必要的 VLM 调用。
5. 已采用成本统计面板，展示调用次数、缓存命中率和成本节省比例。

### 3. 未最终采用或仅作为备选的措施

1. 更复杂的本地离线模型替代方案：可行但开发与部署成本较高，暂未作为主方案。
2. 更激进的多级缓存策略：有潜力进一步降本，但实现和调试成本较高。
3. 长期记忆存储：对体验有帮助，但会带来额外复杂度与存储成本，暂未重点引入。

## 三、项目总结

本项目当前已经完成了从输入、理解、推理到输出的完整闭环，并通过缓存、摘要和规则路由等方式控制了模型调用成本。整体上，系统已经具备较好的演示能力和工程完整性，同时也保留了后续优化空间，尤其是 Avatar 视觉表现和语音播放兼容性方面。由于时间和token成本原因，部分文档没有同步更新，很抱歉。
